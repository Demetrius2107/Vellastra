## 第零部分 项目README总览

### 0\.1 项目定位

星垂野内容系统是一套**业务与架构同步演进**的后端实战项目，以「个人博客→CMS内容管理→全场景UGC知识社区」的业务迭代为载体，同步完成「单体架构→模块化单体→微服务分布式架构」的技术升级。
项目拒绝一次性过度设计，每个阶段的功能复杂度与架构难度严格匹配，开发者可以小步快跑式落地，每一步都能获得对应的技术成长，无需中途推翻重构。

### 0\.2 三阶段演进路线

|阶段|业务形态|架构模式|核心成长目标|
|---|---|---|---|
|阶段一|个人博客 \+ 星垂野多节点联邦体系|Spring Boot单体架构|掌握后端基础开发规范、RBAC权限、缓存、对象存储、分布式接口通信与幂等性设计|
|阶段二|多角色CMS \+ 轻量文字社区|模块化单体 \+ 中间件拓展|掌握复杂业务模块化拆分、读写分离、异步消息队列、全文检索、缓存一致性等中型项目核心能力|
|阶段三|类知乎全场景UGC知识社区|Spring Cloud微服务分布式架构|掌握分布式事务、分库分表、服务治理、高可用容灾、推荐/广告中间件设计等大厂核心技术栈|

### 0\.4 设计核心理念

1. **渐进式无重构**：下一阶段完全复用上一阶段的代码与表结构，仅做拓展与升级，核心逻辑无需重写

2. **生产级细节**：所有接口与表设计均参考大厂生产规范，覆盖异常处理、边界场景、风控安全等玩具项目缺失的环节

3. **面试导向**：每个核心场景都对应后端高频面试考点，项目落地过程同步完成面试考点的实战理解

4. **联邦特色**：贯穿全项目的星垂野联邦体系，作为分布式能力的练习载体，从入门到进阶逐步升级

### 0\.5 文档结构说明

1. 第一部分：三阶段数据表结构设计初稿，遵循渐进复用原则，含分库分表规则与中间件专属表

2. 第二部分：全量接口清单与生产级设计细节，每个接口包含参数定义、完整业务流程、大厂设计考量、面试考点

3. 第三部分：推荐系统与广告系统中间件组件设计，覆盖架构分层与核心实现逻辑

4. 第四部分：架构演进与面试考点总映射，快速对应技术点与项目场景

##### 接口1：管理员登录

- **接口路径**：`/api/admin/auth/login`

- **请求方式**：POST

- **请求参数**

|参数名|类型|必填|校验规则|说明|
|---|---|---|---|---|
|username|string|是|3\-32位字母数字组合|用户名|
|password|string|是|6\-64位|明文密码，传输层HTTPS加密|
|captcha|string|否|4位字符|图形验证码，连续失败3次后强制必填|

- **响应数据**

```JSON
{
  "code": 200,
  "data": {
    "access_token": "xxx",
    "refresh_token": "xxx",
    "expires_in": 7200,
    "user_info": {
      "id": 1,
      "username": "admin",
      "nickname": "管理员",
      "avatar": "xxx"
    }
  }
}
```

- **完整业务流程**

    1. 参数格式校验，不合法直接返回参数错误

    2. 风控校验：查询IP登录失败次数，连续失败3次强制校验验证码，连续失败5次锁定IP15分钟

    3. 查询用户账号，校验账号是否存在、是否被禁用

    4. BCrypt算法比对密码，密码错误则累计失败次数，返回对应提示

    5. 密码校验通过后，清空该账号的登录失败计数

    6. 生成双JWT令牌：access\_token有效期2小时（接口鉴权用），refresh\_token有效期7天（刷新令牌用），令牌与用户绑定存入Redis

    7. 记录登录日志：登录IP、设备UA、登录时间、登录结果

    8. 异地登录检测：比对常用登录地，异常登录触发邮件提醒

- **大厂设计细节**

    - 密码错误统一返回“用户名或密码错误”，不泄露账号存在性，防止撞库

    - 令牌存储Redis，支持主动踢人下线、单点登录控制

    - 登录失败次数按IP\+账号双维度计数，兼顾防暴力破解与用户体验

- **面试考点与设计考量**

    1. 为什么采用双JWT令牌机制？和单令牌相比有什么优势？

    2. JWT有什么优缺点？怎么实现JWT的主动失效？

    3. 防暴力破解有哪些常见方案？各自的优缺点是什么？

    4. BCrypt加密为什么比MD5、SHA更适合存密码？

##### 接口2：新增文章

    - 操作日志记录变更前后快照，支持操作回溯

- **请求方式**：GET

#### 3\.1\.1 架构定位

#### 3\.1\.2 分层架构

1. **数据采集层**

    - 采集方式：业务埋点 \+ MQ异步上报（用户曝光、点击、点赞、停留时长等行为）

    - 存储：行为日志写入Kafka，同步到HDFS离线存储、Doris实时分析

2. **离线计算层**

    - 技术栈：Spark \+ 离线调度

    - 核心任务：每日计算用户画像、协同过滤模型训练、热门榜单统计、内容特征提取

3. **实时计算层**

    - 技术栈：Flink

    - 核心任务：实时更新用户兴趣标签、实时热点内容计算、实时行为反馈修正推荐权重

4. **召回层（多路召回）**

    - 热门召回：全局热门、分类热门、话题热门，解决冷启动

    - 内容召回：基于内容标签相似度召回（TF\-IDF、余弦相似度）

    - 协同过滤召回：基于用户的协同过滤、基于物品的协同过滤

    - 关注召回：用户关注的创作者最新内容

5. **排序层**

    - 粗排：轻量模型快速过滤，缩小候选集

    - 精排：LR/GBDT模型，综合点击率、点赞率、停留时长等特征打分排序

6. **重排层**

    - 规则过滤：过滤已读、违规、低质内容

    - 打散策略：同分类、同创作者内容打散，避免同质化

    - 业务干预：置顶内容、广告插入位预留
